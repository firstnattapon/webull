Webull API Overhaul — Five-Step Planner

ยกเครื่องเส้นทางซื้อขายให้ยึดความสามารถของ Manual Test Lab และสัญญา API ทางการ โดยถือว่า “ส่งคำสั่งสำเร็จ” ยังไม่เท่ากับ “ได้รับการ fill และ position เปลี่ยนจริง”

สถานะ ณ ตอนนี้: regression tests ผ่านออฟไลน์ 276 tests และ UAT จริงผ่านครบทั้ง read-preview, MARKET full-fill/position reconciliation และ LIMIT cancel roundtrip; ผลนี้ไม่ใช่หลักฐานว่า production หรือเงินจริงผ่าน

1. Goals — นิยามผลสำเร็จ

ความถูกต้องของ order

ทุก order ต้องมีตัวตนที่ตรวจย้อนกลับได้ด้วย client_order_id/order_id, สถานะ terminal ที่เชื่อถือได้ และยอด fill สะสมที่ไม่ถอยหลัง

ความถูกต้องของ position

ประกาศ ORDER_FILLED ได้ต่อเมื่อ position_before, position_after และ delta ตรงกับ fill ภายใน epsilon 1e-6

Acceptance gate

ต้องผ่าน UAT read/preview, market place และ limit place/cancel ครบ 100% ก่อน commit/push ส่งงาน

2. Problems — ช่องว่างเดิม

  • เคยถือว่า response จาก place คือผลสำเร็จ ทั้งที่ order อาจยัง pending, partial, cancelled หรือ failed
  • การค้นหา order และ open-order guard อาจอ่านเพียงหน้าแรกหรือยอมรับ response หลวมเกินไป
  • quote เคยเสี่ยง fallback ไปใช้ราคาปิด ซึ่งไม่ใช่ราคาปัจจุบันสำหรับ safety gate
  • สถานะระหว่าง invocation ไม่แข็งแรงพอ ทำให้เสี่ยงส่งซ้ำหรือประกาศ fill ก่อน position reconcile
  • ค่า environment, endpoint, account และ mutation intent ต้องผูกแบบ exact เพื่อป้องกันยิงผิดบัญชี

3. Diagnosis — สัญญา API ที่ใช้จริง

ใช้ SDK ทางการเพียงเส้นทางเดียว: account_v2 สำหรับ account/balance/positions, Market Data สำหรับ quote และ order_v3 สำหรับ preview/place/open/history/detail/cancel โดยตั้ง endpoint ก่อนสร้าง client และตรวจ schema แบบ fail closed

Operationรูปทรง response ที่ยอมรับ
Account listtop-level array; รายการมี account_id
Positionstop-level array; รายการมี symbol และ quantity
Quote snapshottop-level array; รายการมี symbol และ price โดยไม่ fallback เป็น close
Previewobject ที่มี estimated_cost และ estimated_transaction_fee
Place / Cancelflat object ที่ผูกกับ client_order_id และ order_id

Detailtop-level group object เดียวที่มี orders[]; ไม่ค้นแบบ recursive ใน wrapper ที่ไม่อยู่ในสัญญา

4. Design — Manual Test Lab 10 operations

เริ่มจาก authenticated connection ที่ endpoint/region/account ตรงกัน แล้วใช้ flow ต่อไปนี้เป็นแนวทางหลัก 100%:

  1. Account list และยืนยัน account identity
  2. Balance
  3. Positions
  4. Quote snapshot
  5. Market order preview
  6. Market order real submission
  7. Open orders แบบ paginate ทุกหน้า
  8. Order history แบบ paginate ทุกหน้า
  9. Order detail
  10. Cancel order และยืนยันสถานะ CANCELLED/CANCELED

Durable lifecycle ข้าม invocation

PREPARED → SUBMISSION_UNKNOWN | SUBMITTED → PARTIAL_FILLED → FILLED | CANCELLED/CANCELED | FAILED

  • บันทึก lifecycle แบบ deterministic ด้วย identity ของ client order; terminal state เป็น absorbing
  • filled_quantity เพิ่มได้อย่างเดียว และการส่ง place/cancel เป็น at-most-once
  • หาก process จบหรือ timeout จะ reconcile pending order จาก detail → open → history ใน invocation ถัดไปก่อนอนุญาต order ใหม่
  • สถานะ fill ยังไม่เสร็จจนกว่า position_reconciled=true และ delta ของ position ตรงกับยอด fill

Safety gates

  • กำหนด WEBULL_ENV ชัดเจน; endpoint, region, API version, account fingerprint, strategy, symbol และ storage identity ต้องตรงแบบ exact
  • preview เป็นข้อบังคับ; quote ต้องเป็น field price; page size, session และ client ID ต้องอยู่ในขอบเขตทางการ
  • ตรวจ open orders ครบทุกหน้าและ fail closed เมื่อ schema ผิด, duplicate หรือ response คลุมเครือ
  • mutation ต้อง arm ด้วย account/order binding แบบ exact และมี notional cap
  • cancel ได้เฉพาะ order ID ที่ test run นั้นสร้างเอง และเรียก cancel ได้ไม่เกินหนึ่งครั้ง
  • ไม่บันทึก key, secret, token หรือ raw API response ลง source, state, test artifact หรือ log
