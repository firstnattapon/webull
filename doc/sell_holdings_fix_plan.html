2 Problems — ช่องว่างที่ขวางอยู่

P1 — qty คงที่แม้ SELL ออก 3 ครั้ง severity 5/5 · gap: คาดลด ~0.31 หุ้น แต่ลด 0

qty ดึงสดจาก Webull ทุก tick (get_position_and_price) — Webull เองรายงานตำแหน่งเดิม แปลว่าออร์เดอร์ไม่ได้ทำให้ตำแหน่งเปลี่ยน

P2 — log บอก ORDER_SUBMITTED ทั้งที่ไม่มีออร์เดอร์จริง severity 4/5

สถานะถูก hard-code ในโค้ด ไม่ได้ยืนยันจาก Webull — ทำให้ dashboard ดูเหมือน “ขายแล้ว” ทั้งที่ไม่มีอะไรเกิด

P3 — ด่าน has_open_order ไม่เคยกันการยิงซ้ำ severity 3/5

ถ้ามีออร์เดอร์ค้างจริง tick ถัดไปต้องได้ PASS_OPEN_ORDER แต่กลับได้ ORDER_SUBMITTED ซ้ำ → ยืนยันว่า “ไม่มีออร์เดอร์ค้าง = ไม่มีออร์เดอร์จริง”

P4 — ไม่รู้ว่าบอทยิงเข้าสภาพแวดล้อมไหน severity 5/5

โค้ดตั้งค่าเริ่มต้น WEBULL_ENV = "uat" (กล่องทราย) และ log ไม่บันทึกว่ายิงเข้า endpoint ไหน

3 Diagnosis — รากของปัญหา

RC1 — บอทยิงเข้า UAT sandbox ไม่ใช่ production น้ำหนัก w = 0.50

  1. ทำไม qty ไม่ขยับ? → ออร์เดอร์ไม่ได้เข้า order book จริง
  2. ทำไมไม่เข้าจริง? → endpoint เป็น UAT (th-api.uat.webullbroker.com) ที่ “รับ” ออร์เดอร์แต่ตำแหน่งเป็นค่าคงที่
  3. ทำไมเป็น UAT? → config.py ตั้ง default WEBULL_ENV="uat" ถ้าไม่ตั้ง env var จะยิงกล่องทราย
หลักฐาน: qty สแตติกเป๊ะ 5 ตำแหน่งทศนิยม + ORDER_SUBMITTED สำเร็จทุกครั้ง = พฤติกรรมคลาสสิกของ sandbox · ต้องยืนยันด้วยค่า env จริงของ deployment

RC2 — โค้ดตี “สำเร็จ” โดยไม่ตรวจว่ารับจริง น้ำหนัก w = 0.35

  1. place_market_order คืน OrderResult โดยไม่เช็คว่ามี order_id / สถานะไม่ใช่ reject
  2. main.py เขียน status:"ORDER_SUBMITTED" ตายตัว ไม่ว่า Webull ตอบอะไร
  3. ผล 200 ที่ไม่มีออร์เดอร์ (reject เศษหุ้น หรือ echo ของ sandbox) เลยกลายเป็น “ส่งแล้ว” หมด

RC3 — ออร์เดอร์เศษหุ้น (~0.1 หุ้น) อาจถูกปฏิเสธ upstream น้ำหนัก w = 0.15

  1. จำนวนสั่ง 0.10247 / 0.10363 / 0.10777 เป็น fractional market order
  2. บาง broker/บัญชีปฏิเสธเศษหุ้น หรือมี min — แต่ พิสูจน์ไม่ได้ เพราะ log เดิมไม่เก็บเหตุผลจาก Webull

4 Design — มาตรการ (จัดอันดับด้วย π = Δ/c)

อันดับมาตรการยิงที่Δ (ผลที่ปิด gap)c (ต้นทุน)π = Δ/cสถานะ
1ยืนยันว่า Webull “รับ” ออร์เดอร์จริง (เช็ค order_id + สถานะ) และเก็บเหตุผลเมื่อ rejectRC2สูงต่ำ (โค้ด)สูงสุด✔ ทำแล้ว
2log สถานะจริง: ORDER_SUBMITTED เฉพาะเมื่อรับจริง ไม่งั้น ORDER_REJECTED + reasonRC2สูงต่ำ (โค้ด)สูง✔ ทำแล้ว
3บันทึก broker_environment/is_production/endpoint ลง log ทุกออร์เดอร์RC1สูงต่ำ (โค้ด)สูง✔ ทำแล้ว
4ตั้ง WEBULL_ENV=prod ใน deployment (ถ้าต้องเทรดจริง) แล้วยืนยันผ่าน /healthRC1สูงมากต่ำ (config)สูงผู้ใช้ต้องยืนยัน
5รัน live test อ่าน order history/positions ด้วย SDK เพื่อดูเหตุผล reject จริงRC1/RC3สูงสูง (egress ถูกบล็อกในนี้)กลางบล็อกโดย network policy
6ถ้าเศษหุ้นถูกปฏิเสธ → ปัดเป็นจำนวนเต็ม หรือใช้ notional order / เพิ่ม min-notional guardRC3กลางกลางกลางรอข้อมูลจากมาตรการ 5

ถ้า ORDER_REJECTED → อ่าน reason แล้วเลือกมาตรการ 6ผู้ใช้/โค้ดรอบวนรอ

สมการวัดผล & ประเมินผล

Goal attainment (อัตรารับจริง)A = ออร์เดอร์ที่ accepted / ออร์เดอร์ที่ตั้งใจส่ง (เป้า A → 1)
ความสอดคล้องของตำแหน่งΔqty_actual = qty(n−1) − qty(n) ควรใกล้ จำนวนสั่ง(SELL)
ความจริงของ logtruth = #(ORDER_SUBMITTED ที่มี order_id จริง) / #(ORDER_SUBMITTED) (เป้า = 1)

🔁 Control loop — เมื่อยังไม่บรรลุเป้าหมาย

สมการควบคุม (error)e_n = Δqty_expected − Δqty_actual (ต่อออร์เดอร์ SELL)
อัตราการลู่เข้าr_n = |e_n| / |e_(n−1)|

เงื่อนไขหยุด: ถ้า |e_n| ≤ ε = 0.02 หุ้น → ออร์เดอร์มีผลจริง บรรลุเป้า · ถ้าไม่ → ไต่ ladder:

สถานการณ์แปลว่าวนกลับไป
r_n < 1 (ช่องว่างหด)แนวทางถูก ต้องทำซ้ำStep 5 · deploy prod แล้ววัดอีก tick
r_n ≈ 1 (นิ่ง — log ยัง REJECTED)รากปัญหาไม่ใช่ env แต่เป็นเศษหุ้นStep 3 · ขยับน้ำหนักไป RC3 → มาตรการ 6
r_n > 1 (แย่ลง/มี error ใหม่)ระบุปัญหาผิดStep 2 · อ่าน reason จาก Webull ใหม่
หลายรอบยังไม่ขยับบัญชี/สิทธิ์เทรดไม่พร้อมStep 1 · ทบทวนว่าบัญชีนี้เทรดเศษหุ้น US ได้ไหม
