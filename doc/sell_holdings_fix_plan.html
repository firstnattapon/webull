แก้ “จำนวนถือครอง (หุ้น) ไม่อัปเดต”

แผน Five-Step สำหรับ webull และ Webull_Dashboard ยึดหลักว่า order ที่ FILLED ยังไม่ใช่หลักฐานว่า holdings เปลี่ยน ต้องรอ Webull Positions ยืนยัน delta และต้องไม่ส่ง order ใหม่ระหว่าง reconciliation

Offline implementation เสร็จแล้ว เอกสารยังไม่เป็น complete จนกว่า credential เดิมถูก revoke/rotate, live acceptance ผ่าน และติดตามหลัง deploy ครบ 24 ชั่วโมง

1. Goals — ผลลัพธ์ที่ต้องพิสูจน์

Production correctness

ทุก filled lifecycle ต้องมี position_reconciled=true และ |position_after − expected_position_after| ≤ 1e-6 ก่อนเป็น terminal

UAT truthfulness

ถ้า shared/simulated Positions ไม่ยืนยัน ให้แสดง PENDING, UNAVAILABLE, MISMATCH หรือ unverified โดยไม่กล่าวว่า holdings เปลี่ยนจริง

Safety

ไม่มี DNA step หรือ order ใหม่ขณะ fill รอ position reconciliation และไม่มี corrective/replacement order อัตโนมัติ

Security

ช่อง Account ID, App Key และ App Secret เริ่มว่าง ไม่มี credential literal และ credential ที่เคยเปิดเผยต้องถูก revoke/rotate

G_target = 1.0 · q_error = |position_after − (position_before + side_sign × filled_quantity)| ≤ 0.000001

3. Diagnosis — สาเหตุราก

RC1 · lifecycle ปิดเร็ว

Positions อ่านไม่ได้ → terminal=true → state ล้าง pending → invocation ถัดไปไม่ reconcile exact order → dashboard ค้างยอดเดิม

RC2 · retry ไม่ durable

HTTP 504 หลัง bounded retry ภายใน invocation เคยจบ flow โดยไม่มี post-fill snapshot ที่ยืนยันแล้ว

RC3–RC4 · contract ไม่แยก provenance

market_state_quantity ไม่บอก before/expected/confirmed/freshness ทำให้แยก upstream lag จาก UI bug ไม่ได้

RC5–RC6 · หลักฐานไม่พอ

Order history พิสูจน์ fill ได้ แต่สร้าง holdings ไม่ได้ และ shared UAT ไม่ใช่หลักฐาน Production

RC7 · credential exposure

credential literals ถูก commit จึงต้อง rotate; blank defaults และ tests ป้องกันการย้อนกลับ แต่ไม่ทดแทนการ rotate

หลักไม่เปลี่ยน: จำนวนถือครองที่ยืนยันต้องมาจาก Webull Positions เท่านั้น ค่า expected มีไว้เพื่อวินิจฉัย ไม่ใช่ verified holdings หรือ realized cash

4. Design — contract และ control loop ใหม่

Durable lifecycle

FILLED execution → ORDER_*_POSITION_PENDING → CONFIRMED → terminal

  • unavailable หรือ delta mismatch คง pending_order และ retry ใน invocation ถัดไปก่อน timestamp/market/DNA gates
  • position_reconcile_cycles ใช้ alert/manual resolution เท่านั้น ห้ามปลด safety gate
  • terminal ได้เมื่อ full/terminal partial fill ยืนยัน Positions แล้ว หรือ terminal order ที่ filled_quantity=0
  • เพิ่ม execution_terminal, position_sync_status, position_observed_at และ manual_resolution_required

Firestore และ Dashboard

Fieldความหมายใช้เป็น holdings?
pre_order_market_state / position_beforesnapshot ก่อน order; immutable ในเชิงความหมายแสดงคอลัมน์ “ก่อน order”
expected_position_afterค่าคาดเพื่อเทียบ deltaไม่ใช้
position_afterWebull Positions หลัง orderใช้เมื่อมี observation จริง
latest_market_statequantity, price, source=webull_positions, observed_at และ reconciliation flagใช้เป็นยอดล่าสุดพร้อม provenance
legacy unavailable/unconfirmedข้อมูลเก่าที่พิสูจน์ย้อนหลังไม่ได้แสดง LEGACY_UNVERIFIED

Safe legacy repair

เครื่องมือ repair เป็น dry-run โดยปริยาย ไม่เรียก Webull และไม่สร้าง holdings จาก history: requeue ได้เฉพาะ filled legacy row ล่าสุดที่ context ครบและไม่มี later fill; แถวเก่าที่กำกวมถูกระบุ LEGACY_UNVERIFIED การอ่าน Positions สดเกิดใน bot invocation ถัดไปหลัง requeue

Measurement

G_n = reconciled filled lifecycles whose dashboard matches / filled lifecycles expected to reconcile

unsafe_progressions = new_orders_while_position_pending + expected_values_presented_as_verified = 0

A = G_n / G_target; ผ่านเมื่อ A ≥ 1 และ unsafe_progressions = 0

Feedback loop

  1. r_n < 1: reconcile exact order รอบถัดไปโดยคง safety gate
  2. r_n ≈ 1: กลับ Step 3 ตรวจ endpoint, parser, freshness และ Firestore lifecycle
  3. r_n > 1: กลับ Step 2 ตรวจ environment, subsequent fills และ data-source identity
  4. ยังห่างหลังหลายรอบ: คง correctness target, แจ้ง manual resolution และห้ามสร้าง quantity หรือส่ง order ใหม่
Stop condition: เอกสารเปลี่ยนเป็น complete ได้เมื่อ credential เดิมถูก rotate, offline/UAT/Production gates ผ่าน, dashboard อัปเดต document เดิมจาก Positions จริง และ monitoring 24 ชั่วโมงพบ G_n=1, q_error≤1e-6, unsafe_progressions=0
